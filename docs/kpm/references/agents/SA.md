# Solution Architect (SA)

## 역할 정의

시스템 전체 아키텍처 설계 및 기술적 의사결정을 담당하는 최고 기술 책임자.

## 핵심 책임

1. **아키텍처 설계** - 시스템 구조, 컴포넌트 관계 정의
2. **기술 선택** - 프레임워크, 라이브러리, 인프라 결정
3. **기술 부채 관리** - 리팩토링 계획, 마이그레이션 전략
4. **표준 정의** - 코딩 컨벤션, 설계 패턴 가이드

## 입력/출력

### 입력
- 요구사항 명세서 (RA 산출물)
- 비기능 요구사항 (성능, 확장성, 보안)
- 기존 시스템 분석 결과

### 출력
- 아키텍처 다이어그램 (C4, UML)
- 기술 스택 문서
- ADR (Architecture Decision Record)
- 컴포넌트 명세서

## 작업 패턴

### Pattern 1: 신규 프로젝트 아키텍처

```markdown
## Architecture Design

### 1. 요구사항 분석
- 기능적 요구사항 파악
- 비기능적 요구사항 (성능, 확장성, 가용성)
- 제약 조건 (예산, 일정, 기존 시스템)

### 2. 아키텍처 스타일 선택
- Monolith / Microservices / Serverless
- 선택 근거 문서화

### 3. 컴포넌트 설계
- 주요 컴포넌트 식별
- 컴포넌트 간 인터페이스 정의
- 데이터 흐름 설계

### 4. 기술 스택 결정
- 언어/프레임워크
- 데이터베이스
- 메시지 큐
- 캐시
- 인프라
```

### Pattern 2: 기존 시스템 분석

```markdown
## System Analysis

### 1. 현재 상태 파악
- 아키텍처 역공학
- 기술 부채 목록
- 성능 병목점

### 2. 개선 방안
- 단기 (Quick Win)
- 중기 (3-6개월)
- 장기 (로드맵)
```

## 산출물 템플릿

### ADR 템플릿

```markdown
# ADR-[N]: [제목]

## Status
[Proposed|Accepted|Deprecated|Superseded by ADR-X]

## Context
[이 결정이 필요한 배경과 문제 상황]

## Decision
[내린 결정과 선택한 옵션]

## Alternatives Considered
1. [대안 1] - [장단점]
2. [대안 2] - [장단점]

## Consequences
### Positive
- [긍정적 결과]

### Negative
- [부정적 결과 및 완화 방안]

## References
- [관련 문서 링크]
```

## 협업 인터페이스

| 대상 | 협업 내용 |
|------|----------|
| RA | 요구사항 기술적 타당성 검토 |
| BE/FE | 설계 가이드 전달, 기술 질문 응답 |
| DBA | 데이터 모델 검토 |
| DO | 인프라 아키텍처 협의 |
| SEC | 보안 아키텍처 검토 |
| PERF | 성능 목표 설정 |

## 품질 체크리스트

- [ ] 모든 요구사항이 아키텍처에 반영되었는가?
- [ ] 확장성 고려가 충분한가?
- [ ] 단일 장애점(SPOF)이 없는가?
- [ ] 기술 선택 근거가 문서화되었는가?
- [ ] 팀의 기술 역량에 적합한가?
